# key_manager

Cryptographic key derivation and management for the access control system.

## Overview

This module derives all cryptographic keys from a single master secret using HKDF (HMAC-based Key Derivation Function). It provides domain separation for different key uses and supports key versioning for secure rotation.

## Domain Separation

Each derived key uses unique HKDF parameters to ensure cryptographic isolation:

| Key Type | Salt | Info String | Purpose |
|----------|------|-------------|---------|
| AEAD Key | `reader_id + key_version` | `"access-aead-v1"` | Frame encryption per-reader |
| Card Pepper | `key_version` | `"card-pepper-v1"` | Card ID hashing |
| Audit HMAC | `"audit-chain-salt-v1"` | `"audit-hmac-v1"` | Audit log integrity |

## Usage Example

### Loading master key and creating KeyManager

```cpp
#include <key_manager/key_manager.hpp>

// Load master key from file
auto masterKey = key_manager::loadMasterKeyHexFile("secrets/master_key.hex");

// Configure key versioning
key_manager::KeyManagerConfig cfg;
cfg.currentKeyVersion = 2;
cfg.allowPreviousKeyVersion = true;  // Accept v1 during rotation

key_manager::KeyManager km(masterKey, cfg);
```

### Deriving keys

```cpp
// Derive AEAD key for reader 42, key version 2
auto aeadKey = km.deriveAeadKey(42, 2);

// Derive card pepper for version 2
auto pepper = km.deriveCardPepper(2);

// Derive audit HMAC key (version-independent)
auto auditKey = km.deriveAuditHmacKey();
```

### Checking key version acceptance

```cpp
uint32_t frameKeyVersion = frame.header.key_version;

if (!km.isAcceptedKeyVersion(frameKeyVersion)) {
    // Reject: key version too old or invalid
}
```

## Security Notes

- **Master Key Storage**: The master key file should have restricted permissions (e.g., `chmod 600`) and be excluded from version control.
- **Key Rotation**: Set `allowPreviousKeyVersion = true` during rotation to accept both old and new keys, then disable after all readers are updated.
- **Domain Separation**: Each key type uses distinct HKDF salt/info to prevent key reuse across different cryptographic contexts.
- **No Key Caching**: Keys are derived on-demand; consider caching if performance is critical.
