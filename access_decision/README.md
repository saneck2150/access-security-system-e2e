# Access Decision Module

The `access_decision` module is the core policy enforcement engine of the access control system. It processes encrypted frames from card readers, validates access requests against role-based policies, and produces allow/deny decisions with full audit logging.

## Components

### DecisionEngine (`engine.hpp`)

The central orchestrator that ties together all components:

- **Frame Processing**: Delegates to `access_core::FrameHandler` for decryption and validation
- **Policy Enforcement**: Checks role-based access control rules
- **Audit Logging**: Records every decision with full context
- **Event Publishing**: Optional real-time events via `runtime_events::EventBus`

```cpp
DecisionEngine engine(&store, hasher, &auditLog, keyManager);
auto result = engine.handleFrameBytes(frameData, replayWindows);
if (result.allow) {
    unlockDoor();
}
```

### IAccessStore (`access_store.hpp`)

Abstract interface for policy storage:

| Method | Description |
|--------|-------------|
| `roleForCardHmac()` | Lookup card's role by HMAC hash |
| `isAllowed()` | Check if role can access door |
| `currentKeyVersionForReader()` | Get reader's current key version |
| `upsertReader()` | Register/update reader |
| `isReaderAllowedDoor()` | Check reader-door authorization |

Implementations:
- `SqliteAccessStore` (access_storage) - Production persistent storage
- `AccessPolicy` (this module) - Simple in-memory for testing

### CardIdHasher (`card_id_hasher.hpp`)

Privacy-preserving card ID hashing:

```cpp
auto pepper = keyManager.deriveCardPepper(currentVersion);
CardIdHasher hasher(pepper);
std::string hmac = hasher.hmacHex("04:A1:B2:C3:D4");
// Store hmac in database, not raw card UID
```

Key features:
- Uses HMAC-SHA256 with derived pepper key
- Supports key rotation (lookup with current + previous pepper)
- Card UIDs never stored in plaintext

## Decision Reasons

| Reason | Description |
|--------|-------------|
| `ok` | Access granted |
| `parse_error` | Frame structure invalid |
| `replay` | Replay attack detected |
| `decrypt_failed` | Decryption or MAC failed |
| `bad_payload` | JSON payload malformed |
| `unknown_reader` | Reader not registered |
| `unknown_card` | Card HMAC not in store |
| `forbidden` | Role not allowed for door |
| `bad_action` | Action not "open" |
| `no_store` | Store not configured |

## Key Rotation Support

The engine supports transparent key rotation:

1. Reader sends frame encrypted with key version N
2. Engine validates using key version N
3. Card HMAC is computed with current pepper (version N)
4. If card not found AND `allowPreviousKeyVersion` is enabled:
   - Recompute HMAC with previous pepper (version N-1)
   - Lookup again in store
5. This allows gradual migration without re-enrolling cards
