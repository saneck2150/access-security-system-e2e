# runtime_events

Thread-safe event bus for real-time monitoring and logging of access control system activity.

## Overview

This module provides a publish-subscribe mechanism for runtime events. Components throughout the system push events to the bus, and clients (such as the admin web UI) poll for new events to display real-time activity logs.

## Event Structure

```cpp
struct Event {
    uint64_t id;          // Auto-assigned unique ID
    uint64_t ts_unix_ms;  // Timestamp when event occurred
    
    std::string kind;     // Category: "frame", "decision", "audit", "admin"
    std::string message;  // Human-readable description
    
    uint32_t reader_id;   // Reader that triggered the event
    uint32_t door_id;     // Associated door
    uint64_t seq;         // Frame sequence number
    
    bool allow;           // Access granted (true) or denied (false)
    std::string reason;   // Result reason: "ok", "replay", "unknown_card", etc.
};
```

## Usage Example

### Publishing events

```cpp
#include "runtime_events/event_bus.hpp"

runtime_events::EventBus bus(1024);  // Keep last 1024 events

runtime_events::Event e;
e.ts_unix_ms = now_ms();
e.kind = "decision";
e.message = "Access granted to door 5";
e.reader_id = 1;
e.door_id = 5;
e.seq = 42;
e.allow = true;
e.reason = "ok";

uint64_t id = bus.push(std::move(e));
```

### Polling for new events

```cpp
uint64_t lastSeen = 0;

// Poll loop (e.g., in HTTP handler)
auto events = bus.getAfter(lastSeen, 200);
if (!events.empty()) {
    lastSeen = events.back().id;
    // Process/return events...
}
```

### Getting the latest ID

```cpp
uint64_t currentId = bus.lastId();
// Useful for clients to initialize their cursor
```

## Thread Safety

- All `EventBus` methods are **thread-safe** via internal mutex
- Multiple producers can call `push()` concurrently
- Multiple consumers can call `getAfter()` concurrently
- `push()` and `getAfter()` can be called simultaneously from different threads

## Memory Behavior

- Fixed maximum capacity (default 1024 events)
- When full, oldest events are automatically evicted (FIFO)
- Event IDs are monotonically increasing and never reused
- Clients can detect gaps in IDs to know events were dropped
