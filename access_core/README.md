# Access Core Module

The `access_core` module handles encrypted frame processing from card readers. It provides decryption, replay attack detection, key version validation, and timestamp checks.

## Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                      FrameHandler                            │
├─────────────────────────────────────────────────────────────┤
│  handle(frameBytes)                                          │
│       │                                                      │
│       ├─ parseFrame()          → Parse binary frame          │
│       ├─ currentKeyVersionForReader() → Validate reader      │
│       ├─ isReaderAllowedDoor() → Check reader-door binding   │
│       ├─ Key version check     → Current or previous         │
│       ├─ isReplay()            → Sliding window check        │
│       │                                                      │
│       └─ tryDecrypt()                                        │
│              │                                               │
│              ▼                                               │
│         FrameDecryptor                                       │
│              ├─ AEAD decrypt with header as AAD              │
│              └─ Timestamp validation (optional)              │
│                                                              │
│       ▼                                                      │
│  HandleResult { allow, reason, plaintext, header }           │
└─────────────────────────────────────────────────────────────┘
```

## Components

### FrameHandler (`handle_frame.hpp`)

Main entry point for frame processing. Orchestrates all validation steps.

```cpp
FrameHandler handler(keyManager, replayWindows, &store, config);
auto result = handler.handle(frameBytes);
if (result.allow) {
    // Process result.plaintext
}
```

**Configuration options:**

| Option | Default | Description |
|--------|---------|-------------|
| `antiReplayEnabled` | `true` | Enable replay attack detection |
| `replayWindowSize` | `256` | Sliding window size for replay detection |
| `maxCtLen` | `4096` | Maximum ciphertext length |
| `maxSkewMs` | `0` | Max timestamp skew (0 = disabled) |
| `allowPreviousKeyVersion` | `true` | Accept previous key version for rotation |
| `enforceReaderDoorBinding` | `true` | Verify reader is authorized for door |

### FrameDecryptor (`frame_decryptor.hpp`)

Low-level AEAD decryption with optional timestamp validation.

```cpp
crypto_lib::aead::SecureAead aead(key);
FrameDecryptor decryptor(aead, {.maxSkewMs = 60000});
auto result = decryptor.decrypt(frame);
```

## Result Codes

| Code | Description |
|------|-------------|
| `ok` | Frame valid, plaintext available |
| `frame_too_small` | Frame smaller than minimum size |
| `bad_magic` | Invalid frame magic bytes (not "AS01") |
| `bad_protocol_version` | Unsupported protocol version |
| `ciphertext_too_large` | Ciphertext exceeds maxCtLen limit |
| `frame_truncated` | Frame ends before expected |
| `frame_trailing_bytes` | Unexpected data after frame |
| `parse_error` | Generic parse failure |
| `no_store` | Access store not configured |
| `unknown_reader` | Reader ID not registered |
| `reader_door_forbidden` | Reader not authorized for door |
| `bad_key_version` | Key version mismatch |
| `replay` | Sequence already seen |
| `key_derivation_failed` | Failed to derive AEAD key |
| `mac_verification_failed` | AEAD tag/MAC invalid |
| `decrypt_failed` | Generic decryption failure |
| `too_old` | Timestamp too far in past |
| `too_future` | Timestamp too far in future |

## Replay Attack Protection

The handler maintains a sliding window of seen sequence numbers per reader:

1. New frames are checked against the window
2. If sequence was seen before → reject as replay
3. On successful decryption → remember sequence in window
4. Window automatically evicts old entries

## Key Rotation Support

When `allowPreviousKeyVersion` is enabled:

- Frames with current key version are accepted
- Frames with `currentVersion - 1` are also accepted
- Allows gradual migration without service interruption

## Dependencies

- `key_manager` - AEAD key derivation
- `crypto_lib` - AEAD encryption (ChaCha20-Poly1305)
- `protocol_lib` - Frame parsing and replay window
- `access_decision` - IAccessStore interface
